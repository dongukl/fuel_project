  - CAP_POSITION_OFFSET (x,y,z) 오프셋 상수 추가, 최종적으로 z=-0.01로 확정

  - 문 닫을 때 위치를 하드코딩 값이 아니라 ArUco 카메라로 측정한 위치
    (locked_door_center)를 쓰도록 변경

[트러블슈팅] 문 열림 감지 방법 변경
-----------------------------------------------------------------
- 시도 1: 도어 각도(current_door_angle_deg())가 정체(settle)되는 것을 감지해서
  "다 열렸다"고 판단하려 했음.
- 문제 발견: 이 씬에서 current_door_angle_deg()가 비정상적인 값(예: -141.8도)을
  반환함 (RevoluteJoint axis-angle 표현이 wrap되는 문제로 추정). 각도 기반 정체
  감지가 오작동하여 신뢰 불가.
- 해결: 각도 측정을 포기하고, DOOR_OPEN_FIXED_WAIT_SECONDS(5초) 고정 대기 후
  "다 열렸다"고 가정하는 방식으로 전환. 이후 ArUco door lock으로 실제 위치를
  한 번 더 확정.

[트러블슈팅] 문 닫기 애니메이션이 0도로 바로 스냅되는 회귀 버그
-----------------------------------------------------------------
- 원인: 위치 소스를 ArUco 측정값으로 바꾸는 작업을 하면서, build_close_cover_sequence()에
  reference_angle_deg라는 파라미터를 잘못 추가하고 "동적으로 측정한 열림 각도"를
  애니메이션 시작점으로 쓰도록 구조를 바꿔버림. 그런데 측정값(locked_door_angle_deg)
  자체가 부정확해서 애니메이션이 120→75→30→0도로 보간되지 못하고 바로 0도로 튐.
- 사용자 피드백: "위치값만 aruco마커에서 받아서 쓰는것이었는데 시뮬레이션 자체가
  바뀌면 안되지" → 위치 소스만 바꾸고 애니메이션/시뮬레이션 로직 자체는 건드리지
  말아야 한다는 원칙 확인.
- 해결: build_close_cover_sequence()를 원래 시그니처(reference_angle_deg 파라미터
  제거)로 완전히 되돌리고, COVER_START_DEG/COVER_OPEN_DEG 리터럴 상수를 그대로 사용.
  위치 소스(door_reference_point/center_reference_point)만 locked_door_center로 교체.

[트러블슈팅] ArUco lock 게이트가 계속 거부됨 ([LOCK REJECT] outside gate)
-----------------------------------------------------------------
- 원인: marker_to_door_xyz 오프셋이 world.usda에 저장된 "닫힘(rest pose) 상태"의
  fuel_door 좌표를 기준으로 계산되어 있어서, 카메라로 아무리 다시 측정해도 항상
  "문이 닫혀 있었다면 여기"라는 닫힘 등가 좌표가 나옴 (회전을 전혀 반영하지 못함).
  그런데 게이트 기준값(gate reference)은 "열린 상태"로 90도 회전시킨 추정값을
  쓰고 있어서, 카메라 측정값(닫힘 등가)과 게이트 기준(열림 추정)이 구조적으로
  어긋나 있었음. 어떤 허용오차(tolerance)를 줘도 해결 불가능한 근본 원인.
- 해결: _expected_door_open_position() 같은 "열림 상태로 회전시키는" 헬퍼를 완전히
  제거. 게이트 기준값과 폴백값 모두 task.door_world_position(닫힘 등가 좌표)을
  그대로 사용하도록 수정 — 카메라가 실제로 측정하는 값과 같은 기준으로 비교하게 함.

[트러블슈팅] 덮개 자동 열기 속도가 항상 10으로 덮어써짐
-----------------------------------------------------------------
- 원인: on_play_reset()이 매번 DOOR_AUTO_OPEN_DRIVE_VELOCITY(하드코딩 상수=10)로
  속도를 복원했음. USD 씬에서 속도를 50으로 바꿔도 Play를 누를 때마다 코드가
  10으로 되돌려버림.
- 해결: _setup_door_drive()에서 USD에 실제 설정된 targetVelocity 값을 읽어서
  task.door_open_drive_velocity_from_usd에 캐싱. on_play_reset()은 이 캐시된
  값을 복원하도록 변경 (코드 상수는 캐시 실패 시의 폴백으로만 사용).


[2] 차량 슬라이드-인 시뮬레이션 추가
-----------------------------------------------------------------
- 요청: x축 +10m 지점에서 시작해 실제 주차 위치까지 차가 들어오는 시뮬레이션 추가.
  바퀴는 안 움직여도 됨. 매번 같은 자리가 아니라 x,y축 ±0.2m 오차를 두고 랜덤하게
  주차되도록 해서 일반화 테스트 가능하게.
- 사전 조사:
  - ss.usda에서 사용자가 car_visual + aruco_vehicle_marker를 /World/car라는 Xform
    부모로 묶어둔 것을 확인 — 이 prim 하나만 옮기면 차체+마커가 항상 같이 움직임
    (마커를 "스티커처럼" 붙이는 효과).
  - 기존에 문 회전에 쓰던 set_prim_world_matrix() 패턴을 그대로 재사용 가능함을 확인.
  - SEARCH_GATE_HALF_EXTENT(x/y=0.35m)가 ±0.2m 주차 오차를 충분히 허용한다는 것을
    확인 (ArUco lock 게이트가 깨지지 않음).
  - _discover_fuel_objects()가 차가 도착하기 전(authored 위치)에 한 번 호출되지만,
    이 값은 게이트 기준/타임아웃 폴백용일 뿐 실제 로봇 타겟은 항상 라이브 ArUco
    측정값을 쓰므로 재호출 불필요하다고 판단.
- 구현 (multi_robot_oiling.py):
  - 상수 추가: CAR_ROOT_PRIM_NAME, CAR_ARRIVAL_START_OFFSET(+10m, x축),
    CAR_ARRIVAL_SPEED_MPS(2m/s), CAR_ARRIVAL_TOLERANCE_M, CAR_PARK_XY_NOISE_M(0.2m)
  - CarArrivalController 클래스 신설: 매 Play마다 주차 위치에 ±0.2m 랜덤 오차를
    새로 뽑고, 차를 (목표+10m) 지점에서 목표 지점까지 직선 kinematic 이동시킴.
  - main() 메인 루프에서 car_controller가 끝날 때까지 runner_a/runner_b.tick()을
    호출하지 않고 car_controller.tick()만 실행 — 기존 A/B 로봇 상태머신은 전혀
    수정하지 않고 "시작 시점만 늦추는" 방식으로 통합.
